## 3. 模板接入规则

### 3.1 标签选型规则

这是本次最关键的沉淀，后续不要再混用：

- 普通段落章节：使用 `{%p if %}` / `{%p endif %}`
- 表格行内容：使用 `{%tr if %}` / `{%tr endif %}`

错误示例：

```text
{%tr if has_luji %}
二、路基专业
......
{%tr endif %}
```

如果“二、路基专业”所在位置本质上是普通段落而不是表格行，上面这种写法就可能在 docxtpl 预处理后形成错误的 Jinja 结构。

正确示例：

```text
{%p if has_luji %}
二、路基专业
......
{%p endif %}
```

### 3.2 本次真实根因

这次 `TemplateSyntaxError: Unexpected end of template` 的根因不是简单“少了一个 `endif`”，而是：

- `has_common`
- `has_luji`
- `has_suidao`
- `has_qiaoliang`
- `has_zhanchang`

这 5 个新增块本来位于普通段落，但模板里误写成了 `"{%tr if ... %}"`，导致 docxtpl 在 XML 预处理后出现控制块错位，看起来像缺 `endif`，实际上是标签类型错误引发的外层块未闭合。

## 4. 模板排障 SOP

### 4.1 优先排障顺序

遇到 Word/docxtpl 报错时，按下面顺序处理：

1. 运行 [dump_jinja2.py](file:///d:/code/y20_QoSGenerator/dump_jinja2.py)，导出预处理后的 `jinja_source2.xml`
2. 运行 [analyze_jinja.py](file:///d:/code/y20_QoSGenerator/analyze_jinja.py)，检查 `if/endif` 栈
3. 再结合模板位置判断：
   - 是真正缺标签
   - 还是标签类型用错
   - 还是嵌套跨层导致闭合错位

### 4.2 为什么不能只看 Word 里的 `endif`

因为 Word 里的可视化内容不等于 docxtpl 实际收到的 Jinja 结构：

- 标签可能被拆到多个 run
- 标签所在 XML 层级可能和肉眼看到的不一致
- 有 `endif` 也可能只是关掉了内层块，外层仍未闭合

结论：必须以 `jinja_source2.xml` 和标签栈分析结果为准。

## 10. Word 模板标签拆分问题排查经验（2026-07-15）

### 10.1 问题现象

在站前专业-桥梁专业勾选"有桥梁疏散通道视频"后，生成的 Word 文档中**没有输出"桥梁疏散通道视频杆基础"部分**。后端代码逻辑正确，前端提交数据格式正确，但渲染结果始终缺失该章节。

### 10.2 根因分析

问题出在模板文件的 XML 层面。Word 在保存 `.docx` 时，会因为**拼写检查标记（`<w:proofErr>`）**将 Jinja2 标签的文本拆分为多个 `<w:r>` 中的 `<w:t>` 片段。

例如 `{%p if has_qlsstdsp %}` 在原始 XML 中实际结构为：

```xml
<w:r>
  <w:t>{%p if </w:t>
</w:r>
<w:proofErr w:type="spellStart"/>
<w:r>
  <w:t>has_q</w:t>
</w:r>
<w:r>
  <w:t>lsstdsp</w:t>
</w:r>
<w:proofErr w:type="spellEnd"/>
<w:r>
  <w:t> %}</w:t>
</w:r>
```

`docxtpl` 的 `patch_xml` 方法在合并这些拆分片段时**不稳定**——有时能正确合并，有时会丢失 `{%p` 段落级前缀，导致条件判断退化为内联 `{% if %}`，在特定环境下无法正确渲染对应章节。

受影响的标签共 9 个：

| 标签 | 拆分片段 |
|---|---|
| `{%p if has_qlsstdsp %}` | `{%p if ` + `has_q` + `lsstdsp` + ` %}` |
| `{%p if has_fangjian %}` | `{%p if ` + `has_fangjian` + ` %}` |
| `{%p if has_dianli %}` | `{%p if ` + `has_` + `dianli` + ` %}` |
| `{%p if has_nuantong %}` | `{%p if ` + `has_` + `nuantong` + ` %}` |
| `{%p if has_qianyin %}` | `{%p if ` + `has_` + `qianyin` + ` %}` |
| `{%p if has_jiechuwang %}` | `{%p if ` + `has_` + `jiechuwang` + ` %}` |
| `{%p if has_wuxian %}` | `{%p if ` + `has_` + `wuxian` + ` %}` |
| `{%p if has_jixie %}` | `{%p if ` + `has_` + `jixie` + ` %}` |
| `{%p if has_cheliang %}` | `{%p if ` + `has_` + `cheliang` + ` %}` |

### 10.3 修复方法

编写脚本从 **`<w:r>` 边界**替换被拆分的标签，将多个 `<w:t>` 片段合并为单个：

```python
# 核心思路：从 <w:r> 到 </w:r> 截取 run_block，
# 提取所有 <w:t> 文本，合并为单个 run
run_block = content[r_start:r_end]
text_fragments = re.findall(r'<w:t[^>]*>([^<]*)</w:t>', run_block)
if len(text_fragments) > 1:
    merged_text = ''.join(text_fragments)
    merged_run = f'<w:r>{first_rpr}<w:t xml:space="preserve">{merged_text}</w:t></w:r>'
    result = result[:r_start] + merged_run + result[r_end:]
```

修复后 `{%p if has_qlsstdsp %}` 变为单个连续 `<w:t>` 文本，不再被拼写检查标记拆分。

### 10.4 排查经验总结

1. **不要只检查后端代码和前端数据**——即使两者都正确，模板 XML 层面的问题同样会导致渲染异常
2. **直接检查 `.docx` 中的 `word/document.xml`**，用 `zipfile` 提取后搜索标签文本片段
3. **关注 `<w:proofErr>` 标签**——Word 拼写检查是拆分 Jinja 标签的主要元凶
4. **修复时从 `<w:r>` 边界替换**，不要从文本位置替换，否则会破坏 XML 结构
5. 修复后需要验证：`has_xxx=True` 时内容输出、`has_xxx=False` 时内容隐藏、缺失分组时默认值兜底
6. 原始模板应保留备份（`.bak_splitfix`），以便回滚

### 10.5 预防建议

- 在 Word 中编辑模板时，关闭拼写检查再添加 Jinja 标签
- 或使用 `python-docx` 脚本直接操作 XML，避免 Word 自动插入 `<w:proofErr>`
- 每次修改模板后，运行 `python -m unittest test_new_professions_template` 验证
