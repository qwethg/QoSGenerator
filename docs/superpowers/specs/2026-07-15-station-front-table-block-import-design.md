# 施工图互提资料站前三张表块 Excel 导入设计

## 背景

当前“施工图互提资料”页面已经支持：

- 通过 `formData` 提交互提资料参数
- 通过 `station_front` 分组承载站前基础信息
- 由后端 `generate_cross_data_docx()` 先用 `docxtpl` 渲染变量，再用 `python-docx` 做章节重编号

站前专业模板中目前存在 3 个需要按项目实际情况选择性输出的数据表：

- 区间分支引下槽里程表
- 桥上预留引下位置表
- 有线通信过轨里程表

这 3 个对象与“桥梁疏散通道视频”类似，都需要一个开关来控制显示与隐藏，但它们并不是简单布尔项，而是带结构化表格数据的“表块（table block）”。

本次已确认的前提如下：

- 用户手动导入 Excel
- 三张表各自使用独立 Excel 文件
- Excel 采用固定列模板
- 导入状态即开关状态，不再额外设置单独勾选框
- 未导入时，对应表块整块不输出
- 已导入时，对应表块整块输出，并用 Excel 数据覆盖 Word 中现有表格

## 目标

- 在“站前基础信息”卡片中，为 3 个站前表块提供独立的 Excel 导入能力。
- 继续沿用 `station_front` 分组组织前后端数据，避免顶层字段膨胀。
- 将 3 个对象按“表块”而非“裸表格”处理。
- 实现“有导入数据则整块显示、无导入数据则整块删除”的统一逻辑。
- 前端直接解析 Excel 为 JSON，后端继续收 JSON，不引入新的文件上传接口。
- 后端使用 `python-docx` 对 Word 模板中的现有表块做定点填充与删除，避免在 `.docx` 模板中增加复杂表格循环标签。
- 为用户提供“下载 Excel 模板”能力，降低手工整理格式的成本。

## 非目标

- 本次不改造为后端接收 `multipart/form-data` 文件上传。
- 本次不把表前说明文字交给用户在 Excel 中维护。
- 本次不重构现有 12 个专业章节开关逻辑。
- 本次不把所有站前项抽象为通用低代码配置引擎。
- 本次不修改原有非这 3 个表块之外的章节正文。

## 方案对比

### 方案 A：后端直接接收 Excel 文件并在模板中用 Jinja 表格循环

前端上传文件，后端解析 Excel，再在 Word 模板中使用 `{%tr for %}` 等标签控制表格行输出。

优点：

- Excel 解析集中在后端。
- 理论上模板驱动程度更高。

缺点：

- 需要把现有 JSON 提交流程改为文件上传流程。
- Word 模板表格循环标签容易引入 XML 层级和标签闭合问题。
- 与本项目已出现过的模板嵌套风险高度重合，维护成本高。

### 方案 B：前端解析 Excel 为 JSON，后端仍通过 Jinja 表格循环填充

前端读 Excel 并把行数据转成 JSON，再交由后端用 `docxtpl` 在表格中循环输出。

优点：

- 不需要新增文件上传接口。
- 前端可以先做格式校验。

缺点：

- 仍然依赖 Word 模板表格循环。
- 一旦后续要控制整块删除，模板标签会快速变复杂。
- 仍然存在 `docxtpl` 表格标签与 Word XML 结构冲突的风险。

### 方案 C：前端解析 Excel 为 JSON，后端用 `python-docx` 定点填充现有表块

前端把 Excel 转为 JSON 并提交，后端保持 `docxtpl` 的现有文本变量渲染，再在渲染后的文档中用 `python-docx` 精确定位并处理这 3 个表块。

优点：

- 完全复用现有 JSON API，改动范围可控。
- 把复杂度从脆弱的 Word 模板标签转移到 Python 代码，稳定性更高。
- 更容易实现“整块显示/整块删除”。
- 与当前项目已验证的“渲染后再做文档级处理”模式一致。

缺点：

- 需要编写一层表块定位、删块和填表逻辑。

## 最终方案

采用方案 C：

- 前端使用 SheetJS 在浏览器侧解析 Excel
- 解析结果写入 `station_front.imported_tables`
- 后端继续接收 JSON
- `docxtpl` 负责现有布尔/文本变量渲染
- `python-docx` 负责三张表块的定位、删除和数据覆盖

## 设计原则

### 1. 以“表块”作为控制单位

本次控制的不是单一表格，而是一个完整表块。表块至少包含：

- 表标题
- 紧邻标题或表格的说明文字
- 表格本体
- 紧邻表格的注释文字

因此最终规则是：

- 未导入 Excel：整个表块删除
- 已导入 Excel：整个表块保留，并用导入数据覆盖表格内容

### 2. 固定文案留在模板，结构化数据来自 Excel

叙述性文字、标题、注释仍保留在 Word 模板中统一维护，不进入 Excel。

Excel 只承载用户需要维护的结构化数据行。

这样可以保证：

- 模板排版统一
- 用户无需维护固定说明文字
- 文档输出不会因为 Excel 文案差异而风格漂移

### 3. 删除范围必须显式配置，避免误删章节正文

实现时不采用“按前后若干段自动猜测”的模糊删除策略，而是为每个表块显式定义：

- 标题锚点
- 与标题绑定的前后说明段
- 与表格绑定的表后注释段

只有被明确纳入表块配置的段落才会一并删除，防止误删上方正常专业说明。

## 前端设计

### 数据结构

在现有 `formData.station_front` 下新增 `imported_tables` 分组：

```js
station_front: {
  has_qlsstdsp: false,
  imported_tables: {
    interval_branch_downlead: {
      enabled: false,
      file_name: '',
      row_count: 0,
      rows: []
    },
    bridge_reserved_downlead: {
      enabled: false,
      file_name: '',
      row_count: 0,
      rows: []
    },
    cable_crossing_mileage: {
      enabled: false,
      file_name: '',
      row_count: 0,
      rows: []
    }
  }
}
```

说明：

- `enabled` 由导入状态自动驱动
- `file_name` 用于页面展示与调试
- `row_count` 用于页面提示
- `rows` 为最终提交给后端的结构化数据

### 页面位置

继续放在“站前基础信息”卡片中，与 `has_qlsstdsp` 同组管理。

卡片中分成两类内容：

- 站前布尔开关项
- 站前表块导入项

### 交互形式

每个表块显示为一张独立导入卡片，包含：

- 表块名称
- 简短说明
- 当前状态：未导入 / 已导入
- 已导入文件名
- 已识别行数
- `下载模板` 按钮
- `导入/替换 Excel` 按钮
- `清空` 按钮

### 状态规则

- 用户选择 Excel 并校验通过后：
  - 写入 `rows`
  - `enabled = true`
  - 更新 `file_name` 和 `row_count`
- 用户点击清空后：
  - 清空 `rows`
  - `enabled = false`
  - `file_name = ''`
  - `row_count = 0`
- 若重新导入：
  - 直接覆盖旧数据

### Excel 解析方式

前端使用 SheetJS 读取本地 `.xlsx` 文件。

浏览器侧读取流程采用：

1. 从 `<input type="file">` 读取文件
2. 使用 `file.arrayBuffer()` 获取二进制数据
3. 调用 `XLSX.read(arrayBuffer)` 读取工作簿
4. 读取首个工作表
5. 用 `XLSX.utils.sheet_to_json()` 转为对象数组
6. 进行列头校验、空行过滤和数据标准化

之所以采用浏览器侧解析，是因为当前项目没有构建器，继续走前端 JSON 提交比改造后端文件上传更稳。

### SheetJS 接入方式

由于项目当前通过 `templates/index.html` 直接加载浏览器 ES Module，而不是通过 Vite/Webpack 等构建器管理依赖，本次推荐在页面中通过 CDN 引入 SheetJS 浏览器版本，让 `XLSX` 以全局对象形式可用。

这样可以：

- 不引入新的打包流程
- 直接在现有原生 JS 结构中使用
- 兼顾模板下载和 Excel 读取两种能力

### 模板下载方式

下载模板不依赖仓库中的静态二进制 `.xlsx` 文件，而是在前端根据列定义动态生成 Excel：

- 通过统一列配置生成表头
- 可附带 1 行示例数据或空行
- 调用 SheetJS 导出工作簿并触发下载

这样可以避免把二进制模板文件放进仓库，也能保证模板列定义与前端校验规则始终一致。

## Excel 模板规则

### 通用规则

- 每个表块一个独立 `.xlsx`
- 默认只读取第一个工作表
- 第一行为固定列头
- 列头必须完全匹配
- 空行自动忽略
- 若任一必填列缺失，则前端阻止导入并提示

### 序号策略

Word 表格中的“序号”列由系统按导入顺序自动生成，不要求用户在 Excel 中维护。

这样可以：

- 减少用户填写负担
- 避免序号重复或断号
- 保证重新导入后序号连续

## 后端设计

### 数据读取

后端在 `generate_cross_data_docx(data)` 中继续按现有方式读取：

```python
station_front = data.get('station_front', {}) or {}
imported_tables = station_front.get('imported_tables', {}) or {}
```

对每个表块做安全兜底：

```python
table_payload = imported_tables.get('interval_branch_downlead', {}) or {}
enabled = safe_bool(table_payload.get('enabled', False))
rows = table_payload.get('rows', []) or []
```

### 渲染分层

渲染继续分成两层：

1. `docxtpl` 负责现有普通变量渲染
2. `python-docx` 负责三张表块的后处理

这样可以避免把结构化表格数据塞进 `docxtpl` 模板循环中。

### 表块配置

后端需要建立一份显式表块配置，例如：

```python
TABLE_BLOCK_CONFIG = {
    'interval_branch_downlead': {
        'title': '区间分支引下槽里程表',
        'table_index_hint': 1,
        'prelude_mode': 'title_and_next_note',
        'trailing_mode': 'none'
    },
    'bridge_reserved_downlead': {
        'title': '桥上预留引下位置表',
        'table_index_hint': 2,
        'prelude_mode': 'title_only',
        'trailing_mode': 'none'
    },
    'cable_crossing_mileage': {
        'title': '有线通信过轨里程表',
        'table_index_hint': 3,
        'prelude_mode': 'title_and_next_note',
        'trailing_mode': 'next_note'
    }
}
```

说明：

- `title` 用于定位标题段
- `table_index_hint` 用于辅助校验当前模板中的目标表
- `prelude_mode` 控制标题前后的紧邻说明如何纳入表块
- `trailing_mode` 控制表后注释如何纳入表块

实现时可以根据这份配置精确决定删除边界，而不是靠全文搜索做模糊匹配。

### 当前模板下的表块边界判断

根据当前施工图模板结构，这 3 个表块建议这样定义：

#### 1. 区间分支引下槽里程表块

纳入表块：

- 标题：`区间分支引下槽里程表`
- 紧随标题的说明段：`（区间无线GSM-R基站、直放站引下槽的里程由无线通信专业计列，具体见无线通信过轨预留里程表）`
- 表格本体

不纳入表块：

- 上方“分支引出槽如下图所示：”及其对应图示区域
- 下方“3、过轨”章节正文

#### 2. 桥上预留引下位置表块

纳入表块：

- 标题：`桥上预留引下位置表`
- 表格本体

不纳入表块：

- 上方桥梁引下电缆槽章节正文
- 下方桥梁疏散通道视频章节

#### 3. 有线通信过轨里程表块

纳入表块：

- 标题：`有线通信过轨里程表`
- 紧随标题的说明段：`（区间无线GSM-R基站、直放站过轨已由无线通信专业计列，具体见无线区间设施设置里程表；不包含双线隧道综合洞室过轨里程，双线隧道综合洞室里程见隧道专业综合洞室里程表）`
- 表格本体
- 紧随表格的注释段：`注：当房屋场坪、隧道救援点等位置变化时，过轨和引下槽里程需配合调整。`

不纳入表块：

- 上方桥梁疏散通道视频章节
- 下方站场专业章节

### 删除逻辑

若某表块 `enabled = false` 或 `rows` 为空，则：

1. 定位标题段
2. 根据表块配置找出需一并删除的说明段/注释段
3. 定位并删除对应表格
4. 只删除该表块，保留上下正文

### 填充逻辑

若某表块 `enabled = true` 且 `rows` 非空，则：

1. 保留标题、说明和注释段
2. 清空目标表格除表头外的旧数据行
3. 按导入顺序写入新数据行
4. 第一列序号由系统从 `1..n` 自动生成
5. 对空备注等可选字段写空字符串

### 错误处理

- 若表块配置中的标题未找到，返回明确错误，提示模板结构可能已变化
- 若目标表格未找到或表头列数不匹配，返回明确错误
- 若前端未传 `station_front.imported_tables`，则按“无导入数据”处理，不影响旧功能
- 若 `rows` 不是数组，后端兜底为 `[]`

## 前后端数据流

1. 用户在“站前基础信息”卡片中点击某张表的 `下载模板`
2. 前端按该表列定义生成并下载 Excel 模板
3. 用户填写 Excel 后点击 `导入/替换`
4. 前端解析 Excel、校验列头并转成标准 `rows`
5. 前端将解析结果写入 `station_front.imported_tables.xxx`
6. 用户点击生成文档
7. 后端先用 `docxtpl` 生成基础文档
8. 后端再用 `python-docx` 按表块配置做删除或覆盖
9. 输出最终 `.docx`
