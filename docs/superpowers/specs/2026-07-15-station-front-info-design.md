# 施工图互提资料站前基础信息扩展设计

## 背景

当前“施工图互提资料”页面已经支持按专业开关生成不同章节，并通过前端 `formData` 直接提交给后端 `generate_cross_data_docx()`，由后端组装模板上下文后渲染 Word 模板。

本次需求是在“接收专业配置”之后、“房屋类型配置”之前增加一个“站前基础信息”卡片，用于承载站前相关专业的补充选项。首个落地项为桥梁专业新增的“桥梁疏散通道视频”勾选项，对应模板中的：

```jinja
{%p if has_qlsstdsp %}
桥梁疏散通道视频
{%p endif %}
```

考虑到后续还会持续新增其它站前子项，本次采用分组对象方案，而不是继续将字段平铺在 `formData` 顶层。

## 目标

- 在前端新增“站前基础信息”卡片。
- 当勾选了路基、站场、桥梁、隧道任意一个专业时，显示该卡片。
- 当勾选了桥梁专业时，在卡片中显示“是否有桥梁疏散通道视频”选项。
- 当前端勾选该选项时，后端向模板传入 `has_qlsstdsp = true`，模板输出对应条目。
- 当取消桥梁专业时，自动清空该子选项，避免误输出。
- 为后续继续增加站前子项预留清晰的数据结构。

## 非目标

- 本次不调整现有模板章节结构。
- 本次不重构“房屋类型配置”或“电缆沟与分支槽参数”区域。
- 本次不引入新的通用配置驱动渲染机制。

## 方案选择

### 方案 A：平铺字段

直接在 `formData` 顶层新增 `has_qlsstdsp`。

优点：
- 改动最小。

缺点：
- 后续站前子项越来越多时，顶层字段会持续膨胀。
- 语义归属不清晰，不利于维护。

### 方案 B：站前分组对象

在 `formData` 中新增：

```js
station_front: {
  has_qlsstdsp: false
}
```

优点：
- 结构清晰，便于后续继续扩展站前补充项。
- 前端卡片内字段归属明确。
- 后端只需做一次分组读取和模板变量映射。

缺点：
- 比平铺方案多一层前后端映射。

### 方案 C：配置驱动渲染

将“哪些专业显示卡片、卡片里有哪些子项”抽成配置表统一渲染。

优点：
- 扩展性最好。

缺点：
- 对当前项目复杂度偏高，超出本次需求范围。

## 最终方案

采用方案 B，在前端引入 `station_front` 分组对象，并在后端将其映射为模板可直接使用的顶层上下文字段。

## 前端设计

### 数据结构

在 `MutualDataGenerator.js` 的 `formData` 中新增：

```js
station_front: {
  has_qlsstdsp: false
}
```

后续新增站前补充项时，统一挂载在 `station_front` 下，例如：

```js
station_front: {
  has_qlsstdsp: false,
  has_xxx: false,
  xxx_note: ''
}
```

### 卡片位置

新增“站前基础信息”卡片，位置固定在：

- “接收专业配置”之后
- “房屋类型配置”之前

### 显示条件

当下列任一专业被勾选时，显示“站前基础信息”卡片：

```js
formData.has_luji ||
formData.has_zhanchang ||
formData.has_qiaoliang ||
formData.has_suidao
```

### 子项显示条件

“是否有桥梁疏散通道视频”仅在勾选桥梁专业时显示：

```js
formData.has_qiaoliang
```

### 交互行为

- 勾选桥梁专业后，显示“是否有桥梁疏散通道视频”复选项。
- 取消桥梁专业后，自动将 `formData.station_front.has_qlsstdsp` 重置为 `false`。
- 重新勾选桥梁专业时，默认保持未勾选状态，由用户重新决定。

### 视觉风格

复用当前页面卡片体系，保持与“接收专业配置”“房屋类型配置”一致的视觉语言：

- 使用现有 `card`、`card-header`、`card-body` 结构。
- 选项样式优先复用现有卡片式勾选视觉模式，避免引入新的风格分裂。
- 保持页面留白与信息密度平衡，避免新增区域显得拥挤。

## 后端设计

### 数据读取

后端在 `generate_cross_data_docx(data)` 中读取：

```python
station_front = data.get('station_front', {}) or {}
```

### 模板上下文映射

将前端分组对象中的字段映射为模板直接可用的顶层变量：

```python
has_qlsstdsp = safe_bool(station_front.get('has_qlsstdsp', False))
```

并合并进最终 `context`。

### 兼容性

- 模板继续使用 `has_qlsstdsp`，无需修改当前模板条件写法。
- 若前端未传 `station_front`，则默认按 `False` 处理，不影响已有功能。

## 数据流

1. 用户勾选接收专业。
2. 若命中站前四个专业之一，则页面显示“站前基础信息”卡片。
3. 若勾选桥梁专业，则卡片中显示“是否有桥梁疏散通道视频”。
4. 用户勾选该选项后，前端提交 `station_front.has_qlsstdsp = true`。
5. 后端将其映射为模板变量 `has_qlsstdsp`。
6. Word 模板根据 `{%p if has_qlsstdsp %}` 输出或隐藏“桥梁疏散通道视频”条目。

## 错误处理

- 如果前端未传 `station_front`，后端按空对象处理。
- 如果 `station_front.has_qlsstdsp` 为非布尔值，后端通过 `safe_bool()` 做兜底转换。
- 如果取消桥梁专业，前端主动清空该值，避免后端收到陈旧数据。

## 测试与验证

### 功能用例

1. 勾选桥梁专业，未勾选视频选项：
   - 显示站前基础信息卡片
   - 显示桥梁视频选项
   - 文档不输出“桥梁疏散通道视频”

2. 勾选桥梁专业，并勾选视频选项：
   - 文档输出“桥梁疏散通道视频”

3. 先勾选桥梁和视频，再取消桥梁专业：
   - 桥梁视频选项隐藏
   - `station_front.has_qlsstdsp` 自动重置为 `false`
   - 文档不输出该条

4. 仅勾选路基、站场或隧道之一：
   - 显示站前基础信息卡片
   - 不显示桥梁视频选项

5. 四个站前专业全部未勾选：
   - 不显示站前基础信息卡片

### 回归关注点

- 不影响现有 12 个专业开关逻辑。
- 不影响现有房屋配置、房屋启用状态计算和电缆沟参数提交。
- 不影响旧模板在未传该字段时的默认行为。
